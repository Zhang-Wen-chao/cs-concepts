# Abstraction and Encapsulation - 抽象与封装

> 隐藏复杂性，设计清晰接口。

## 🎯 核心概念

### 抽象 (Abstraction) — 提取关键特征，忽略无关细节
```
汽车抽象：关注 加速/刹车/转向，忽略 发动机内部
```

### 封装 (Encapsulation) — 隐藏内部实现，只暴露必要接口
```
遥控器：暴露 音量/频道按钮，隐藏 电路板/信号处理
```

---

## 🔍 抽象 vs 封装

| 特性 | 抽象 | 封装 |
|------|------|------|
| **关注点** | 做什么 (What) | 怎么做 (How) |
| **目的** | 简化复杂性 | 隐藏实现 |
| **层次** | 设计层面 | 实现层面 |
| **例子** | 接口定义 | 私有变量 |

**关系**：抽象是思想，封装是实现抽象的手段。

## 🏗️ 封装机制

| 机制 | 作用 | 示意 |
|------|------|------|
| **访问控制** | public/protected/private 限定可见性 | `self.__balance = 0` |
| **Property** | getter/setter 包装字段访问 | `@property → setter` |
| **模块化** | 命名空间 + 下划线约定内部 | `_internal() vs public()` |

## 🎨 抽象层次

```
高层: UserManager (业务逻辑) — 依赖 Storage 接口
中层: Storage (抽象接口)
低层: FileStorage / DatabaseStorage (具体实现)
```

**依赖倒置**：高层不依赖低层具体实现，都依赖抽象接口。

---

## 💡 核心设计原则

| 原则 | 含义 | 反例→正例 |
|------|------|-----------|
| **SRP** | 一个类只做一件事 | `User.save+email+report` → 拆分为三个类 |
| **OCP** | 对扩展开放，对修改封闭 | `if type==A → elif type==B` → 多态扩展 |
| **LSP** | 子类可替换父类 | `Square extends Rectangle` 要正确处理 |
| **ISP** | 客户端不需要的接口别强制实现 | `Robot implements Worker(work+eat)` → 拆分接口 |
| **DIP** | 依赖抽象，不依赖具体 | `new EmailNotifier()` → `依赖 Notifier 接口` |

**记住**：
1. 抽象关注"做什么"，封装关注"怎么做"
2. 好的抽象让代码更简单，接口小而精确
3. 依赖抽象而非具体实现
4. 封装保护数据完整性
