# 性能评估 (Performance Evaluation)

> 如何科学地衡量和优化计算机性能

## 🎯 核心概念

**性能 = 1 / 执行时间**。关键问题：如何定义、测量、比较性能？

- **墙上时间 (Elapsed Time)**：程序从开始到结束的总时间，含 I/O 等待、OS 开销
- **CPU 时间**：CPU 实际执行该程序的时间（用户时间 + 系统时间）

---

## 📐 性能方程

```
CPU 时间 = 指令数 × CPI × 时钟周期时间
         = (指令数 × CPI) / CPU 频率
```

| 变量 | 含义 | 影响因素 |
|------|------|---------|
| 指令数 (IC) | 程序执行的指令总数 | 算法、编译器 |
| CPI | 每条指令平均时钟周期数 | CPU 架构、Cache Miss、分支预测 |
| 时钟周期 | 1/频率 | 制程、功耗 |

### 平均 CPI 构成

```
平均 CPI = Σ (CPI_i × 指令比例_i)
```

| 指令类型 | 比例 | CPI |
|---------|------|-----|
| ALU | 50% | 1 |
| Load/Store | 35% | 2 |
| 分支 | 15% | 1~4 (取决预测) |

**典型 CPI ~ 1.5**。影响因素：流水线停顿(+1-3 cycles)、Cache Miss (+10-200)、分支预测失败(+10-20)。

---

## ⚖️ Amdahl 定律

```
加速比 S = 1 / ((1-P) + P/N)
  P = 可并行部分占比
  N = 并行度（核心数）
  (1-P) = 串行部分
```

| 并行比例 | 4核加速 | 16核加速 | 理论极限 |
|---------|---------|---------|---------|
| 50% | 1.60x | 1.88x | 2x |
| 80% | 2.50x | 3.88x | 5x |
| 90% | 3.08x | 5.71x | 10x |
| 95% | 3.48x | 8.42x | 20x |

**启示**：串行部分是瓶颈（即使 90% 可并行，32 核也只有 ~7.5 倍加速）；优先优化热点代码。

## 📊 基准测试

| 类型 | 示例 | 用途 |
|------|------|------|
| SPEC CPU | SPECspeed (单线程) / SPECrate (多线程) | CPU 综合性能 |
| CoreMark | 列表/矩阵/状态机 | 嵌入式 CPU 效率 (CoreMark/MHz) |
| LINPACK | 求解线性方程组 | 超算排名 (FLOPS) |
| 微基准 | Cache 延迟/内存带宽 | 特定硬件指标 |
| 真实应用 | 编译时间、DB TPS、Web QPS | 用户体验 |

## 🔬 性能分析工具

### Linux perf
```
perf stat ./program          # 统计: cycles, instructions, IPC, Cache Miss
perf record/report           # 采样分析热点函数
perf stat -e stalled-cycles-frontend,stalled-cycles-backend  # 流水线效率
```

### Intel Top-Down 分析
```
Level 1:
  Retiring (有效工作)         → 40%
  Bad Speculation (分支错误)  → 10%
  Frontend Bound (取指瓶颈)   → 20%
  Backend Bound (执行瓶颈)    → 30%
    → Memory Bound (L1/L2/L3/DRAM)
    → Core Bound (执行单元)
```

## 🚀 优化优先级

| 层级 | 手法 | 加速潜力 |
|------|------|---------|
| 1. 算法 | O(n²)→O(n log n) | 10-1000x |
| 2. 数据结构 | 链表→数组(Cache友好) | 2-10x |
| 3. 编译器 | -O0 → -O3 | 2-5x |
| 4. 多线程 | 多核并行 | 0.5N-0.8N × |
| 5. SIMD | 标量→向量化 | 2-4x |
| 6. 微架构 | Cache对齐、减少分支 | 10-30% |

### 优化原则
1. **先测量，再优化** — profiler 说了算
2. **优化热点** — 80% 时间花在 20% 代码上
3. **编译器很聪明** — 清晰代码易优化
4. **避免过早优化** — 正确性优先
5. **实测为准** — 理论 ≠ 实际
