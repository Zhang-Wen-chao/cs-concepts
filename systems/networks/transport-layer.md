# Transport Layer - 传输层

> 如何保证端到端可靠传输？TCP 和 UDP 的核心区别是什么？

## 🎯 传输层的作用

```
功能：
1. 端到端通信：进程到进程
2. 可靠传输：确保数据完整有序
3. 流量控制：防止发送方过快
4. 拥塞控制：防止网络过载
5. 复用与分用：多个应用共享网络

数据单位：段 (Segment)
类比：端口号 = 门牌号，TCP = 保价快递，UDP = 普通信件
```

## 🔌 端口号

| 分类 | 范围 | 用途 | 常见端口 |
|---|---|---|---|
| 知名端口 | 0–1023 | 系统服务 | 80(HTTP), 443(HTTPS), 22(SSH), 53(DNS) |
| 注册端口 | 1024–49151 | 应用程序 | 3306(MySQL), 6379(Redis) |
| 动态端口 | 49152–65535 | 客户端临时 | — |

`套接字 = IP 地址 + 端口号`，唯一标识一个通信端点。

## 📊 TCP vs UDP 对比

| 特性 | TCP | UDP |
|---|---|---|
| 连接 | 面向连接（三次握手） | 无连接 |
| 可靠性 | ✅ 可靠（确认、重传、排序） | ❌ 不可靠（可能丢包乱序） |
| 有序性 | ✅ 按序到达 | ❌ 不保证顺序 |
| 流量控制 | ✅ 滑动窗口 | ❌ 无 |
| 拥塞控制 | ✅ 慢启动、拥塞避免 | ❌ 无 |
| 头部开销 | 20–60 字节 | 8 字节 |
| 适用场景 | HTTP, FTP, SSH, 邮件 | DNS, 视频/语音, 游戏, IoT |

UDP 头部 8 字节：`源端口(2B) | 目标端口(2B) | 长度(2B) | 校验和(2B) | 数据`
TCP 头部 20–60 字节：包含源/目标端口、序列号、确认号、标志位(SYN/ACK/FIN/RST)、窗口大小

## 🤝 TCP 连接管理

### 三次握手 (连接建立)

```
客户端                   服务器
  │                       │
  ├── SYN (seq=x) ──────→│  ① 客户端请求连接
  │←─ SYN-ACK (seq=y,ack=x+1) ─┤  ② 服务器同意
  ├── ACK (ack=y+1) ────→│  ③ 客户端确认
  │                       │
  └──────── ESTABLISHED ────────┘
```

> **为什么三次？** 防止过期 SYN 干扰，同时确认双方的发送/接收能力并同步序列号。
> 两次不够（过期 SYN 会错误建立连接），四次多余。

### 四次挥手 (连接关闭)

```
客户端                   服务器
  │                       │
  ├── FIN ──────────────→│  ① 客户端：我要关了
  │←─ ACK ───────────────┤  ② 服务器：知道了
  │      (服务器继续发剩余数据...)   │
  │←─ FIN ───────────────┤  ③ 服务器：我也关了
  ├── ACK ──────────────→│  ④ 客户端：好的
  │                       │
  └── TIME_WAIT (2MSL) ──┘
```

> **为什么四次？** 全双工，每个方向单独关闭；服务器 ACK 和 FIN 不能同时发，中间可能还有数据。
> **为什么要 2MSL？** 确保对方收到最后的 ACK，防止旧连接包干扰新连接。

## 🔄 可靠传输机制

| 机制 | 描述 |
|---|---|
| **序列号 & 确认号** | 每个字节一个序号，接收方回 Ack 确认 |
| **超时重传** | 定时器到期未收到 Ack 则重传 |
| **快速重传** | 收到 3 个重复 Ack 立即重传（不等超时） |
| **滑动窗口** | 控制已发送未确认的数据量，窗口大小由接收方通告 |
| **拥塞控制** | 慢启动 → 拥塞避免 → 快速恢复；通过拥塞窗口 (cwnd) 控制发送速率 |

### 滑动窗口示意

```
发送方窗口: [已确认] [已发送未确认] [可发送] [不可发送]
              ↑ base          ↑ next    ↑ base+win
```
