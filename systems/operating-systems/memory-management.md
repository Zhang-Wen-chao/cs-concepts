# Memory Management - 内存管理

> 操作系统如何管理有限的内存？如何让每个程序都觉得自己拥有整个地址空间？

## 🎯 核心问题

1. **内存不够用** — 虚拟内存让程序使用大于物理内存的空间
2. **多程序隔离** — 进程 A 不能访问进程 B 的内存
3. **碎片化** — 外部碎片让大块分配失败

---

## 🧩 地址空间

```
虚拟地址 (程序看到的)   ──[MMU]──→   物理地址 (实际内存)
每个进程独立 4GB                         所有进程共享
```

### 进程地址空间布局

```
高地址: 内核空间 (OS)
        栈 (向下增长)
        ↑ 空隙 ↓
        堆 (向上增长)
        数据段 (全局变量)
低地址: 代码段 (只读)
```

---

## 🗺️ 内存管理方式

### 分页 (Paging) — 现代方案

将虚拟内存分成固定大小的**页**，映射到物理内存的**页框**，无需连续。

```
虚拟内存:  页0 → 页框2, 页1 → 页框5, 页2 → 页框0 ...
物理内存:  不连续映射，消除外部碎片
```

### 地址转换

```
虚拟地址 = (页号, 页内偏移)  →  查页表 →  物理地址 = (页框号, 页内偏移)
```

### 多级页表

```
问题：32位下每个进程需要 4MB 连续页表
解决：二级/多级页表 → 只为实际使用的页分配页表项，延迟分配
```

---

## 💨 TLB (Translation Lookaside Buffer)

页表查询的硬件缓存，命中率 ~98%。

```
访问流程：查 TLB (几 CPU 周期) → 命中→直接物理地址
                                → 未命中→查页表(几十周期)→更新 TLB
```

**注意**：进程切换时 TLB 需刷新（线程切换不需要）。

---

## 💾 虚拟内存与 Swap

```
物理内存不够 → 把不活跃的页换出到磁盘（Swap 分区）
访问不在内存的页 → 缺页中断 (Page Fault) → OS 从磁盘加载 → 重新执行
```

### 页面置换算法对比

| 算法 | 思路 | 优点 | 缺点 |
|------|------|------|------|
| **FIFO** | 替换最早进入的 | 简单 | Belady 异常（更多帧反而缺页更多） |
| **LRU** | 替换最久未用的 | 性能好 | 硬件实现开销大 |
| **Clock** | 第二次机会 | 接近 LRU，开销低 | 近似算法 |
| **Optimal** | 替换未来最晚用的 | 最优 | 需要预知未来，不可实现 |

**LRU ≈ 最优的实用方案**；Clock 是 LRU 的低成本近似。

---

## 📊 全过程速览

```
CPU 访存 → TLB 命中? → Y→ 物理地址
                 → N→ 查页表 → 页在内存? → Y→ 物理地址 + 更新 TLB
                                     → N→ 缺页中断 → 磁盘加载 → 更新页表
```
