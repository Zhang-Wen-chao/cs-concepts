# Process Synchronization - 进程同步与互斥

> 多个进程/线程如何安全地访问共享资源？如何避免数据竞争与死锁？

## 🎯 核心问题

**数据竞争 (Race Condition)**：多个线程同时读写共享变量，结果取决于指令交错顺序。

```
counter += 1  =  读取 → 计算 → 写回  (非原子!)
线程1读(0) → 线程2读(0) → 线程1写(1) → 线程2写(1)  → 丢了一次更新
```

### 临界区 (Critical Section)

访问共享资源的代码段需满足三个要求：
1. **互斥** — 同一时刻只有一个进程在临界区
2. **进步** — 无进程在临界区时，想进入的必须能进入
3. **有限等待** — 不会无限等待

---

## 🔒 同步原语

### 对比表

| 原语 | 特点 | 主要用途 |
|------|------|----------|
| **互斥锁 (Mutex)** | 二元，谁抢到谁进 | 保护临界区 |
| **信号量 (Semaphore)** | 计数，PV 操作 | 资源池、同步 |
| **条件变量 (CV)** | 等待/通知，需配合锁 | 生产者-消费者 |
| **事件 (Event)** | 一次性的唤醒信号 | 线程间通知 |
| **屏障 (Barrier)** | 等待 N 个线程到齐 | 分阶段并行 |

### 信号量示例

```python
empty = Semaphore(N)   # 缓冲区空位
full  = Semaphore(0)   # 产品数量
mutex = Lock()         # 保护 buffer

producer: empty.P() → lock → put → unlock → full.V()
consumer: full.P()  → lock → get → unlock → empty.V()
```

## 🍽️ 经典同步问题

### 1. 生产者-消费者
缓冲区容量 `N`：用**计数信号量**同步空位和产品数，用**互斥锁**保护缓冲区。

### 2. 读者-写者
**多读者可同时读，写者独占**。用读写锁实现：
```
第一个读者 → 阻止写者进入（上写锁）
最后一个读者离开 → 允许写者进入
写者在临界区 → 阻止一切读者和写者
```

### 3. 哲学家就餐
5 支筷子 = 5 把锁。避免死锁的典型解法：
- **按序拿** — 先拿编号小的筷子
- **限人** — 最多 4 人同时拿筷子（`Semaphore(4)`）

## ☠️ 死锁 (Deadlock)

### 四个必要条件

| 条件 | 含义 |
|------|------|
| **互斥** | 资源不能共享 |
| **持有并等待** | 持有一把锁再等另一把 |
| **非抢占** | 不能强行剥夺 |
| **循环等待** | P1 等 P2，P2 等 P1 的锁 |

### 处理策略

```
预防 → 破坏任一条件（如按锁的 id 排序取锁）
避免 → 银行家算法（不安全则不分配）
检测 → 等待图 + 环路检测
恢复 → 杀掉/回滚进程
```

```python
# 预防循环等待：按全局顺序取锁
locks = [lock1, lock2, lock3]
locks.sort(key=id)
with locks[0]: with locks[1]: ...
```

## 🔓 无锁编程 (Lock-Free)

当锁成为瓶颈时，用**原子操作 (CAS)** 代替锁。

```
CAS(memory, expected, new): 如果*memory==expected → 写new，否则重试
```

| 对比 | 有锁 (Mutex) | 无锁 (Lock-Free) |
|------|-------------|-----------------|
| 实现难度 | 简单 | 复杂 |
| 高竞争性能 | 差（阻塞/切换多） | 好（自旋重试） |
| 死锁 | ❌ 可能 | ✅ 不会 |
| 公平性 | 好 | 不保证 |

**注意**：CAS 有 **ABA 问题**（变量被 A→B→A，CAS 误判未变）→ 加版本号解决。

**实践建议**：大多数场景用锁就够；无锁只在性能热点、经测试证实后使用。

---

## 💡 最佳实践

1. **尽量缩小临界区** — 只保护共享部分的读写
2. **避免嵌套锁** — 若不可避免则按全局顺序取锁
3. **使用上下文管理器** — `with lock: ...` 自动释放
4. **优先使用线程安全容器** — `queue.Queue` 等
